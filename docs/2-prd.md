## 1. 제품 개요

### 1.1 제품 목적

20대~50대 직장인이 업무와 일상의 할일을 체계적으로 관리할 수 있도록, 인증 기반의 개인 맞춤형 할일 관리 웹 서비스를 제공한다. 카테고리 분류와 다양한 필터링 기능으로 우선순위 파악을 쉽게 하고, 어떤 디바이스에서도 동일한 경험을 제공하는 반응형 웹 UI로 구현한다.

### 1.2 핵심 가치

| 가치 | 설명 |
|------|------|
| 접근성 | PC·모바일 브라우저 어디서든 동일한 UI/UX 제공 |
| 신뢰성 | DB 기반 데이터 저장으로 디바이스 전환 시에도 데이터 유지 |
| 개인화 | 사용자 정의 카테고리로 자신만의 분류 체계 구성 가능 |
| 보안 | JWT 기반 인증으로 본인 데이터만 접근 가능 |

## 4. 기술 스택

### 4.1 전체 구성

| 영역 | 기술 | 버전 |
|------|------|------|
| 프론트엔드 | React + TypeScript | React 19 |
| 상태 관리 | Zustand | 최신 안정 버전 |
| 서버 상태 / API 통신 | TanStack Query | 최신 안정 버전 |
| 백엔드 | Node.js + Express | LTS |
| 데이터베이스 | PostgreSQL | 17 |
| DB 클라이언트 | pg (node-postgres) | - |
| 인증 | JWT (JSON Web Token) | - |
| JWT 구현 라이브러리 | jsonwebtoken | 최신 안정 버전 |
| 비밀번호 해시 | bcrypt | 최신 안정 버전 |

### 4.2 아키텍처 구성도

```
[Browser]
   |
   | HTTP/HTTPS (REST)
   v
[Express REST API Server]
   |-- /api/auth     (인증)
   |-- /api/users    (사용자 정보)
   |-- /api/categories (카테고리)
   |-- /api/todos    (할일)
   |
   | pg 라이브러리
   v
[PostgreSQL 17]
```

### 4.3 인증 전략

| 단계 | 방식 | 내용 |
|------|------|------|
| 1차 (현재) | JWT | Access Token 발급, Zustand(메모리)에 저장, HTTP Authorization Header 전달 |
| 2차 (확장) | OAuth 2.0 Social | Facebook, Google 등 소셜 로그인 연동 |

- JWT Access Token은 Zustand `authStore`의 메모리에 저장한다. `localStorage`, `sessionStorage`, `httpOnly Cookie` 사용 금지.
- 페이지 새로고침 시 토큰이 초기화되므로 사용자는 재로그인이 필요하다. (소규모 프로젝트 허용 범위)
- JWT Secret은 환경 변수로 관리하며 소스 코드에 하드코딩하지 않는다.
- OAuth 확장을 고려하여 인증 미들웨어는 전략 패턴(Strategy Pattern)으로 분리 설계한다.

### 4.4 DB 연동 제약

> **필수**: PostgreSQL 연동은 반드시 `pg` (node-postgres) 라이브러리를 사용한다. ORM(Sequelize, TypeORM, Prisma 등) 사용 금지.

### 5.3 기능 상세 요구사항

#### 5.3.1 회원가입 (UC-01)

| 항목 | 내용 |
|------|------|
| 입력값 | 이메일, 비밀번호, 이름 |
| 유효성 | 이메일 형식 검증, 이메일 중복 불가(BR-U-01), 비밀번호 최소 8자 |
| 비밀번호 처리 | bcrypt 해시 후 저장 (BR-U-02) |
| 성공 시 | 로그인 화면으로 이동 |
| 실패 시 | 이메일 중복 오류 메시지 표시 |

#### 5.3.2 로그인 (UC-02)

| 항목 | 내용 |
|------|------|
| 입력값 | 이메일, 비밀번호 |
| 성공 시 | JWT Access Token 발급, 메인(할일 목록) 화면으로 이동 |
| 실패 시 | 이메일/비밀번호 구분 없는 단일 오류 메시지 (보안 목적, 계정 존재 여부 노출 방지) |

#### 5.3.3 할일 목록 조회 및 필터링 (UC-06)

| 필터 | 옵션 | 규칙 |
|------|------|------|
| 카테고리 | 전체 / 카테고리 선택 | BR-F-01 |
| 기간 종료 여부 | 전체 / 기간 내 / 기간 초과 | BR-F-02 |
| 완료 여부 | 전체 / 완료 / 미완료 | BR-F-03 |
| 복수 필터 | AND 조건 동시 적용 | BR-F-04 |

#### 5.3.4 카테고리 (UC-04)

- 기본 카테고리(개인, 업무, 쇼핑)는 시스템 제공이며 수정/삭제 불가 (BR-C-01)
- 사용자 정의 카테고리 추가 가능, 동일 사용자 내 이름 중복 불가 (BR-C-04)
- 1차에서는 카테고리 수정/삭제 UI 미제공 (2차 이관)

#### 5.3.5 개인 정보 수정 (UC-03)

| 항목 | 내용 |
|------|------|
| 수정 가능 필드 | 이름, 비밀번호 (이메일 변경 불가, BR-U-04) |
| 비밀번호 변경 조건 | 현재 비밀번호 입력 후 일치 검증 필수. 불일치 시 변경 거부 |
| 성공 시 | 변경 사항 저장 후 완료 메시지 표시 |
| 실패 시 | 현재 비밀번호 불일치 오류 메시지 표시 |

#### 5.3.6 할일 등록 (UC-05)

| 항목 | 내용 |
|------|------|
| 필수 입력값 | 제목(title), 카테고리(category_id) (BR-T-01, BR-T-02) |
| 선택 입력값 | 상세 설명(description), 종료예정일(due_date) |
| due_date 유효성 | 입력한 경우 오늘 날짜 이후여야 한다. 과거 날짜 입력 시 오류 반환 (BR-T-06) |
| 성공 시 | 할일 저장 후 목록 화면에 즉시 반영 |
| 실패 시 | 필수값 누락 또는 유효하지 않은 due_date 입력 시 오류 메시지 표시 |

#### 5.3.7 할일 수정 (UC-07)

| 항목 | 내용 |
|------|------|
| 수정 가능 필드 | 제목, 설명, 종료예정일, 카테고리 |
| 소유권 검증 | 요청 사용자 소유의 할일만 수정 가능. 타인 소유 접근 시 403 반환 (BR-T-03) |
| due_date 유효성 | 등록 시와 동일하게 현재 날짜 이후여야 한다 (BR-T-06) |
| 성공 시 | 변경 사항 저장 후 목록 즉시 반영 |

#### 5.3.8 할일 완료/미완료 처리 (UC-08)

| 항목 | 내용 |
|------|------|
| 완료 처리 | `is_completed = true`, `completed_at = 현재 일시` 로 저장 (BR-T-04) |
| 미완료 복원 | `is_completed = false`, `completed_at = null` 로 초기화 (BR-T-05) |
| 소유권 검증 | 요청 사용자 소유의 할일만 처리 가능. 타인 소유 접근 시 403 반환 (BR-T-03) |

#### 5.3.9 할일 삭제 (UC-09)

| 항목 | 내용 |
|------|------|
| 삭제 방식 | 소프트 삭제 없이 즉시 영구 삭제 |
| 소유권 검증 | 요청 사용자 소유의 할일만 삭제 가능. 타인 소유 접근 시 403 반환 (BR-T-03) |
| 성공 시 | 삭제 후 목록에서 즉시 제거 |

### 6.4 가용성 및 확장성

| 항목 | 내용 |
|------|------|

### 7.2 페이지 구성

| 페이지 | 경로 | 접근 조건 |
|--------|------|-----------|
| 로그인 | `/login` | 비인증 |

## 8. API 설계 방향

### 8.1 기본 원칙

- RESTful 설계 원칙 준수
- 응답 형식: JSON
- 인증 필요 API: `Authorization: Bearer <token>` 헤더 사용
- 에러 응답: HTTP 상태 코드 + `{ "message": "..." }` 형식 통일

### 8.2 엔드포인트 목록 (예정)

| 메서드 | 경로 | 설명 | 인증 |
|--------|------|------|------|
| POST | `/api/auth/register` | 회원가입 | ❌ |
| POST | `/api/auth/login` | 로그인 | ❌ |
| GET | `/api/users/me` | 내 정보 조회 | ✅ |
| PATCH | `/api/users/me` | 내 정보 수정 | ✅ |
| GET | `/api/categories` | 카테고리 목록 | ✅ |
| POST | `/api/categories` | 카테고리 추가 | ✅ |
| GET | `/api/todos` | 할일 목록 조회 (필터 쿼리 파라미터) | ✅ |
| POST | `/api/todos` | 할일 등록 | ✅ |
| PATCH | `/api/todos/:id` | 할일 수정 | ✅ |
| PATCH | `/api/todos/:id/complete` | 완료/미완료 처리 | ✅ |
| DELETE | `/api/todos/:id` | 할일 삭제 | ✅ |

### 8.3 필터링 쿼리 파라미터 (`GET /api/todos`)

| 파라미터 | 타입 | 예시 | 설명 |
|----------|------|------|------|
| `category_id` | UUID | `?category_id=xxx` | 카테고리 필터 |
| `is_completed` | boolean | `?is_completed=false` | 완료 여부 필터 |
| `overdue` | boolean | `?overdue=true` | 기간 초과 필터 |
